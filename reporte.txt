1. DESCRIPCIÓN GENERAL
------------------------------------------------------------------------------
NemoOS es un sistema operativo hobby para la arquitectura x86 (i386) escrito
desde cero en C++20 y Assembly (GAS/AT&T). Bootea mediante GRUB/Multiboot y
se ejecuta en QEMU. El compilador usado es GNU g++ con flags freestanding
(-ffreestanding -fno-exceptions -fno-rtti). No usa biblioteca estándar;
implementa sus propias funciones libk (memcpy, memset, strcmp, strcpy, itoa).


2. TECNOLOGÍAS UTILIZADAS
------------------------------------------------------------------------------
  Lenguaje (kernel)  : C++20 (-std=c++20, freestanding)
  Lenguaje (low-level): x86 Assembly (GAS, AT&T syntax)
  Build system       : GNU Make
  Toolchain          : g++, as, ld (x86_64-linux-gnu a i386 cruzado)
  Target             : i386 (32-bit, -m32)
  Boot protocol      : Multiboot (GRUB)
  Emulador           : QEMU (qemu-system-i386)
  Linker script      : linker.ld (kernel en 1MB físico)
  Control de versión : Git

    syscalls/
      syscall.cpp/.hpp        - int 0x80 dispatcher

5. ESTADO DE IMPLEMENTACIÓN
------------------------------------------------------------------------------

5.1. Implementado (funcional)
  - Boot multiboot + GRUB ISO
  - Terminal VGA (80x25, scroll, cursor, color, backspace)
  - GDT (6 entradas: null, kernel code/data, user code/data, TSS)
  - IDT (256 entradas, handlers para ISR0, IRQ0, IRQ1, page fault, syscall)
  - PIC 8259 remapeado a ints 32-47
  - Timer PIT 100Hz con contador de ticks
  - Teclado PS/2 (scancode -> ASCII, buffer circular)
  - PMM (gestor de memoria física por bitmap)
  - Paginación (page directory/table, identity map 4MB, CR0.PG)
  - Bump allocator (kmalloc en 16MB, alineado)
  - Kernel panic (mensaje + archivo + línea + halt)
  - Syscalls (int 0x80, syscall 1 = get_ticks)
  - Shell interactivo (help, clear, echo, about, ticks, mem, pages, alloc,
    free, kmalloc, halt, ls, touch, cat, write, mkdir, cd, pwd)
  - RAMFS (128 inodos, directorios, archivos, r/w)
  - Planificador cooperativo (16 slots, funciones como tareas)
  - TSS (para stack de kernel en modo usuario)
  - I/O de puertos (inb/outb/inw/outw/inl/outl)
  - libk (memcpy, memset, strcmp, strncmp, strcpy, itoa)

5.2. Parcial / Stub (incompleto)
  - kmalloc/kfree: kfree es no-op (sin heap manager)
  - Heap: heap.cpp/.hpp completamente vacío
  - VMM (Virtual Memory Manager): vmm.cpp/.hpp vacío
  - VFS (Virtual Filesystem): vfs.cpp/.hpp vacío
  - PMM (separado): pmm.cpp/.hpp vacío (PMM real está en pmm_bitmap)
  - ISR install: isr_install() es función vacía
  - Scheduler: sin preempción (sin context switch por timer)
  - Tasks: sin context switch, sin pilas, sin scheduling por syscall

5.3. No implementado (ausente)
  - Modo usuario (Ring 3) - TSS configurado pero sin switch a user mode
  - Multitarea preemptiva - Timer IRQ no hace context switch
  - Context switching - No hay switch_to / save-restore de tareas
  - Procesos - No hay PCB, PID, ni aislamiento de memoria
  - Cargador de ELF
  - Drivers de disco (ATA/IDE)
  - Sistema de archivos en disco (FAT, ext2, etc.)
  - Networking (no hay stack de red)
  - PCI (no hay enumeración de buses)
  - USB
  - IPC / señales
  - Heap con liberación de memoria
  - VMM (mapeo/desmapeo de páginas, resolución de page faults)
  - RTC/CMOS
  - Mouse PS/2
  - Puerto serial/COM


6. PORCENTAJE ESTIMADO DE COMPLETITUD

                        ~25-30%

                                sin ejecución de procesos)
    Syscalls           :   5%  (1 syscall implementada de las decenas
                                necesarias: fork, exec, open, read, write...)


7. OBSERVACIONES
------------------------------------------------------------------------------
  - No hay README, TODO ni documentación (hasta ahora).
  - No hay .gitignore: los objetos .o, kernel.bin y kernel.iso están
    trackeados en git.
  - Un solo commit ("init commit"): no hay historial de desarrollo.
  - La versión reportada es "NemoOS v0.9" (impresa al bootear).
  - Los comandos del shell están en español/portugués.
  - Varios archivos stub vacíos (heap, vmm, pmm, vfs) indican subsistemas
    planeados pero no iniciados.
  - No hay enmascaramiento de IRQs más allá del remapeo básico del PIC.
